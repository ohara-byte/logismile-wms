## 1. プロジェクト概要

大江ノ郷自然牧場の **倉庫管理システム（WMS）** を新規開発する。
現行WMSはカスタマイズ不可・ダッシュボード/レポート機能なしのため、自社業務に最適化したWebシステムへ刷新する。

### 業務概要

| 項目 | 内容 |
|---|---|
| 用途 | 社内業務システム（自社製造・通過型在庫管理） |
| 対象規模 | 1日 2,000〜3,000件（将来：4,000〜5,000件） |
| 在庫特性 | 通過型（製造 → 出荷）。長期保管なし |
| 基幹連携 | Thomas（基幹）からCSVで出荷指示を受信。1日5〜10回の都度取込 |
| 利用端末 | PC×5 / タブレットHP14×10 / ハンディKEYENCE BT-A500×5 / プリンターSCeaTa CT4-LX×6 |

### 主要業務フロー（サンドイッチ方式）

1. **出荷指示取込**：Thomas CSV → 管理PCからアップロード（1日5〜10回）
2. **検品**：①ピッキング№スキャン → ②商品検品 → ③同梱物確認 → ④納品書№スキャン
3. **QR印刷**：QR印刷フラグONの伝票のみ、QRコード付きラベル（30×40mm）を自動発行
4. **梱包完了**：ステータス → packed

## 4. 認証方式（重要：端末ごとに分離）

### 管理PC（admin / manager）
- **メールアドレス + パスワード**（NextAuth credentials）
- bcrypt ハッシュ化でDB保存
- マスタ管理・CSV取込・レポート・削除/復活操作の権限

### タブレット / ハンディ（staff含む全ロール）
- **★ 社員番号のみ** でログイン（パスワード入力なし）
- `staff.emp_code` を入力 → staff レコード参照 → セッション発行
- 検品作業の即応性優先
- 社内ネットワーク内からのみアクセス可能（イントラネット限定）
- 端末コード（device_code）も合わせて送信し、検品セッションに記録
- 退職者は `staff.active = false` でログイン無効化
- 管理機能（マスタ更新等）には社員番号ログインではアクセス不可

### ロール
| ロール | 主な権限 |
|---|---|
| admin | 全機能（マスタ・ユーザー管理・削除/復活含む） |
| manager | マスタ・CSV取込・レポート閲覧・削除/復活・検品 |
| staff | タブレット/ハンディ検品のみ |

## 5. データモデル（主要概念）

### 商品階層

- **親商品**：受注単位（JANなし）。基幹側にのみ存在し、WMSは直接保持しない
- **構成商品**：WMS連携単位（JAN付き）。検品単位として扱う
- WMS上での親商品表示は「構成品マスタからの逆引き」で補完

### ★ 主キー仕様

| データ | 主キー | 重複可否 |
|---|---|---|
| 出荷伝票 | **ピッキング№** | **★ 重複不可**（重複時は取込エラー） |
| 納品書№ | （非主キー） | **★ 重複可**（ピッキング№配下のため） |
| 構成商品 | 商品コード | 一意 |
| **JANコード** | **（非主キー）** | **★ 重複可**（構成商品コードが異なれば同一JAN許可） |

### ★ 伝票の修正・削除・復活

- **修正**：基幹で新ピッキング№発行 → 旧伝票は管理PCで削除
- **削除**：物理削除せず **論理削除**（`deleted_at` / `deleted_by` / `delete_reason`）
- **復活**：管理PC「🗑 削除済み伝票一覧」から「♻ 復活」ボタンで `deleted_at` をクリア
- **保管期間**：1年保持 → 年次バッチで物理削除（CSVアーカイブ後）
- すべての削除・復活・編集を `order_audit_logs` に記録

## 6. ★ QR印刷（呼称統一）

### 仕様
- プリンター：**SCeaTa CT4-LX × 6台**（社内LAN・固定IP）
- 用紙：**SATO レスプリ・シータラベル 超高感度サーマルB 30×40mm**
- 印字内容：QRコード（**納品書№**をエンコード） + その下に納品書№テキスト併記

### QR印刷フラグ
- 基幹側「熨斗フラグ」を WMS では **「QR印刷フラグ」** と呼称統一（画面・API・帳票すべて）
- 受信値が初期値だが、**検品画面でタップにより手動切替可能**（誤受信や追加対応に対応）
- フラグ ON の伝票のみ、検品完了時に最寄りプリンターへ印刷指示
- フラグ OFF の伝票は印刷をスキップ

### プリンター管理
- **プリンターマスタ**（管理PC）でIP・配置・モデルを一元管理
- **device_printer_map** で端末ごとの「最寄りプリンター」を管理PCから設定
- タブレット/ハンディ側にはプリンター設定画面なし（サーバから既定先を取得）
- 印刷ジョブはサーバ経由でプリンターIP宛てに送信（端末→APIサーバ→プリンター）

## 7. ★ IFアダプタ層（フェーズ1/2 切替対応）

### 設計指針
- 取込処理・データ変換ロジックを **`src/lib/integration/`** に集約
- 共通インターフェース `IntegrationAdapter` を定義
- フェーズ1：`CsvAdapter`（CSV手動アップロード）
- フェーズ2：`ApiAdapter`（基幹API連携・将来）
- WMS本体ロジックはアダプタ差し替えで変更不要

### 共通インターフェース（イメージ）

```typescript
interface IntegrationAdapter {
  importProducts(source: File | ApiConfig): Promise<ImportResult>;
  importShippingOrders(source: File | ApiConfig): Promise<ImportResult>;
}
```

### 項目マッピング
- 設計メモv2.1 表7-1（商品マスタ）／表7-2（出荷指示）に準拠
- 「熨斗フラグ」→「QR印刷フラグ」の読み替えを含む
- マッピングテーブルは `src/lib/integration/mapping.ts` に集約

### CSV取込仕様
- 文字コード：Shift-JIS / UTF-8 自動判定
- 取込頻度：1日5〜10回（都度取込）
- 重複処理：同一ピッキング№はスキップしエラー報告
- JAN検証：空欄／13桁以外／チェックデジット不正 のみエラー（**重複は許容**）

## 8. 検品フロー（サンドイッチ方式）

### タブレット
1. 社員番号ログイン → 端末コード送信
2. **ピッキング№**スキャン → 出荷指示ロード
3. 商品スキャン（JAN/商品コード）→ 数量カウント
4. 強制OK（理由必須）／保留 が必要に応じて
5. 同梱物確認モーダル（のし・パンフ等）
6. 箱選定（固定箱優先／可変箱は容積計算）
7. **★ 納品書№スキャン** → packed ステータス更新
8. **★ QR印刷フラグONの場合のみ** QR印刷を最寄りプリンターへ送信

### ハンディ（KEYENCE BT-A500）
- 連絡事項モーダル（Enterキーで1件ずつ☑）
- のし確認モーダル（Enterキーで確認）
- 検品フローはタブレットと同一API/セッション管理
- スマートフォン最適化UI

### スキャン結果状態
- `matched`（成功・数量加算）
- `over_scan`（数量超過）
- `not_found`（未マッチ）
- `already_done`（既完了）

## 9. 管理PCダッシュボード

### 主要機能
- **全体進捗カード**：当日件数・完了率・残件・平均ペース
- **テーブルグループ別 進捗カード**（7グループ／14テーブル）：進捗率・配置メンバー・終了予定時刻・遅延フラグ
- **ライン・仕分**（独立作業エリア）の進捗
- **段階目標 vs 予測**（時刻別目標との差分）
- **★ 終了予定時刻 自動再計算**：配置メンバー数 × スキル係数 × 残件 × 標準時間
- **★ 遅延警報**：進捗率 -10% 以上 / 直近30分処理不足で自動表示
- **冷凍便・特殊優先表示**
- **連絡事項表示**
- **アラート管理**（未マップ/JAN不備/強制OK/遅延）

### リアルタイム更新
- Server-Sent Events（SSE）：`/api/progress/stream`
- イベント：session_start / scan / complete / hold / assignment_changed

## 10. メンバー割当（Gantt）

### UI仕様
- 縦軸：メンバー（最大40名以上）
- 横軸：9:00〜18:00 の 1時間刻み
- 30分単位スナップでバー配置
- ドラッグ＆ドロップで移動／両端リサイズで時間調整
- ×ボタン削除 / 空きセルクリックで新規追加ピッカー

### プール
- 未設定メンバープール（出勤だが未配置）
- 休みメンバープール（誤割当防止）

### 操作
- 昨日の割当読込
- 朝礼用印刷
- 全クリア
- 保存 → GROUPS.staff へ反映 → 終了予定時刻 自動再計算

## 11. シフト管理

- **シフトマスタ**：720行規模（24名×30日）
- **シフトパターンマスタ**：G7/G6/A7/A6/A4/A3/B7/D7/D6/E6/公休/有休/希休/特休/欠勤 等。編集すると担当者マスタの選択肢に即反映
- **雇用区分マスタ**：正社員A/正社員8h/正社員6h/準社員8h/準社員7h/準社員6h/短時間/嘱託
- **★ GPシフトCSV取込**：4ステップウィザード（ファイル選択→列マッピング自動検出→プレビュー→実行）
- **★ 当日シフトをガントへ反映**：本日出勤者を未設定プールに自動配置・出勤時刻を割当初期値に

## 13. 開発規約

### コード規約
- TypeScript strict モード
- Server Components 優先（'use client' はインタラクティブUIのみ）
- ESLint + Prettier 自動整形
- API Routes は Zod でリクエスト検証
- DBアクセスは Prisma 経由のみ（生SQL禁止、複雑な集計はviewまたはraw SQL関数）

### 命名規則
- ファイル：kebab-case（`order-list.tsx`）
- コンポーネント：PascalCase（`OrderList`）
- 関数・変数：camelCase
- DBテーブル・カラム：snake_case（Prismaで`@@map`/`@map`）

### コミット規約
- `feat: 機能追加`
- `fix: バグ修正`
- `refactor: リファクタリング`
- `docs: ドキュメント更新`
- `chore: 設定変更`

### テスト
- 重要なビジネスロジック（box-selector / jan-validator / IFアダプタ）にはユニットテスト
- E2Eテストは検品フローを最低1ケース
