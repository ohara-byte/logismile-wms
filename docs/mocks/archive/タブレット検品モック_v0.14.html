🎮 デモ操作パネル

以下から代表的な4ケースの伝票読込をシミュレーションできます。v0.14 から開始トリガーは「ピッキング№バーコード」、完了トリガーは「納品書バーコード」のサンドイッチ方式に変更されました。

v0.14 での変更（今回）★現場レビュー反映
・開始トリガーを「納品書バーコード」→「ピッキング№バーコード」に変更（サンドイッチ方式へ移行）
　- 待機画面の文言／フローステップ①／デモパネル説明 を全て更新
　- 待機画面に「※ 梱包完了時は納品書バーコードをスキャン」のサブテキスト追記
・梱包完了「✓ 梱包完了（次へ進む）」ボタンを撤去 → 納品書バーコードスキャンで自動完了（サンドイッチ方式）
　- 最終チェックモーダル内に 📷 納品書バーコード スキャン待ち プロンプトを新設
　- 同梱物・同梱指示・のし が全て☑になると「受付中」（点線→水色枠＋脈動）に切替
　- スキャンすると画面中央に ✓ 梱包完了 トースト表示（1.4秒）→ 印刷判定／完了画面へ自動遷移
　- デモパネルに「📷 納品書スキャン（梱包完了）」ボタン追加
・最終チェックモーダルから「📦 推奨梱包箱」セクションを撤去（左ペインで既に確定済のため重複排除）
　- 検品中ずっと表示されている左ペインの推奨箱はそのまま継続
・検品保留メニューの副題を簡素化：「商品/数量に問題あり
本部に通知して中断」→「スキャン状態を保持して
一時中断する」
　- 通知記載は本部連絡側（右ボタン）に明確に分離
・検品済み／保留中の再スキャン デモ ボタン名を「ピッキング№ 再スキャン」に変更（トリガー一致のため）

v0.13 での変更
・「商品コード」→「構成商品コード」表記に統一（基幹IF認識合わせに基づく）
　- 検品リスト／数量キーパッド／最終チェック画面の見出し
　- 親商品（受注単位／JANなし）と構成商品（出荷単位／JANあり）の階層を明示
　- WMS が受信するのは構成商品レベル。親商品は基幹側にのみ存在

v0.12 での変更
・📦 推奨箱の表示位置を「最終チェックモーダル内」→「検品中の左ペイン（常時表示）」に変更
　- 伝票ロード直後（loadOrder 末尾）から箱候補を表示／検品中ずっと右側で見える
　- ピッキング担当者が箱を準備できるよう、商品検品より前に候補が確定
　- コンパクト縦並びカード（候補名／区分ピル／サイズランク／充填率バー）
・自動展開ロジックを強化：modal が active でない／screenState=inspect／currentOrder 有り の3条件をガード
・最終チェックモーダル内の旧推奨箱セクションを削除（重複防止）
・※将来 基幹側で算出した箱コードを受け取る方式に切替予定（フェーズ2）
　- 現在の WMS 内 suggestBoxes() はあくまで暫定実装
　- 基幹側で `parent_code → box_code` を確定して連携する設計（基幹連携設計メモ v2.0 に追記予定）

v0.11 での変更
・最終チェック画面に「📦 推奨梱包箱」セクション追加
　- 箱マスタ（管理PC v0.15 で管理）から商品構成に応じて候補を自動提案
　- 優先度：①完全一致 専用箱＞②専用箱+追加＞③可変箱（容積/重量/温度帯適合）
　- 上位5件をタブで表示、⭐は推奨候補（スコア最上位）
　- 詳細：寸法・容積・最大重量・温度帯・適合理由・警告（温度帯不一致など）
　- 温度帯推定：商品コードの先頭文字で常温/冷蔵/冷凍 を自動判定（基幹未連携の補助運用）

v0.10 での変更
・検品保留時にスキャン状態を保持（HELD_ORDERS レジストリ）
　- 数量スキャン状況／同梱物チェック／のしチェック／編集内容 を全てスナップショット保存
　- 同じ納品書を再スキャンすると「保留中」ポップアップが表示
　- 「▶ 続きから検品」 で前回の状態を完全復元（途中から再開可能）
　- 「↻ 最初から検品」 で保留状態を破棄して新規検品
　- 「キャンセル」で待機画面に戻る
・本部連絡（📢）も同じ仕組みで保留（指示待ち中も検品済み数量が維持される）
・デモパネルに「⏸ 保留中伝票を再スキャン」ボタン追加（ORDERS[1] が半分スキャン済みで保留される）

v0.9 での変更
・F4ボタン名・色を変更：「⏭ 検品省略」→「📋 一括検品」（btn-success 緑）
・F5 伝票保留にポップアップ追加：従来の即保留 → 「⏸ 検品保留」「📢 本部連絡」 の2択選択
　- 本部連絡：4分類ボタン（🎁のし / 📦商品 / ✏入力内容 / 🌐WEB）+ 補足テキスト
　- 送信先：管理PC「連絡」タブ → 着信バッジで通知
　- ペイロード：from / pkNo / invoice / category / body / sentAt / read（未読フラグ）

v0.8 での変更（再仕様化）
・v0.7 で要件解釈ミスがあったため再仕様化
・商品検品完了で「最終チェック画面を自動展開」（ボタン押下を省略）
　- 全商品スキャン完了で 0.35秒後に自動でモーダルが開く
　- 既に開いている場合や検品済み伝票の再読込時は自動展開しない
・「梱包完了」ボタンを「⏭ 検品省略・梱包完了」用途に転用（オレンジ色 / F4）
　- クリックすると 残数を確認するダイアログ 表示 → OK で全件完了扱い → 自動展開フローへ合流
　- 強制OK と異なり 理由コードは記録しない（検品の流れそのものを省略する用途）
　- 監査ログには「検品省略・梱包完了 残N点を省略」を記録
・強制OK（F2）は別系統として維持（理由コード必須）

v0.7 での変更（取消）
・"梱包完了"ボタンを"📋 最終チェック"にリネームして手動押下を残していた仕様は取消

v0.6 での変更
・梱包完了ボタンを「📋 最終チェック」に名称変更（強制OKと役割分離）
・最終チェック画面に「✓ 商品検品状況」セクション追加：全アイテムの検品状況（X/Y点・残N点・強制OKメッセージ）を一覧表示
・のし関連の同梱物項目を表示から除外（上部のピンクのし表示と重複しないよう "熨斗"/"のし"/"台紙" を含む koudoku/instructions は描画スキップ）
・最終チェックの完了ボタンを「✓ 梱包完了（次へ進む）」に変更（その後 印刷判定→印刷モーダル or 完了画面へ）
・フローステップ表示を「①納品書 → ②商品検品 → ③📋 最終チェック → 完了」に簡素化

v0.6 での変更
・トリガーを「納品書バーコード」に一本化：v0.5までのサンドイッチ検品（①ピッキングNo→②商品→③納品書）を撤廃。納品書1枚で①読み込み →②商品検品 →③同梱物確認 →完了。
・同梱物確認モーダルを新設：商品検品完了後に表示。のし種類・名称（基幹取込データ）と同梱指示・同梱物（WMS補助マスタ／取込データなし）を分けて☑可能。全☑後に「✓ 同梱物 確認完了」で完了画面へ。
・検品済み伝票 再読込ポップアップ：すでに完了済みの納品書バーコードを読むと「✓ 検品済み」と表示。「確認」で待機戻り、「検品戻し（再検品）」で完了状態を解除し再度検品可（操作ログ記録）。
・同梱物の右ペイン表示は「プレビュー」扱いに変更（最終☑はモーダルで実施）。

v0.5 からの継続機能
数量キーパッド「残数入力」方式（残数超過エラー）／特殊梱包フラグ（基幹データ起因）／構成商品のみ表示（親商品は基幹未連携）／連絡事項ポップアップ（起動時☑確認）／QRラベル再発行／強制検品Sticky＋解除バナー。
ハードキー割当（想定）：F1=商品スキャン／F2=強制OK／F3=同梱物ON／F4=📋 一括検品／F5=伝票保留・本部連絡／ESC=中断
ハード構成：タブレット＝HP14系（1280×800想定） ／ 外付スキャナ（USB/Bluetooth HIDモード） ／ ラベルプリンタ＝SATO CT4-LX（熱転写）
動作確認手順（v0.14 サンドイッチ方式）： 起動 → 連絡事項4件☑→「作業開始」 → ケース②（熨斗あり・元セット品）を読込（＝ピッキング№スキャン） → 構成商品3点をスキャン（または🔢ボタン→数量入力）→ 最終チェックモーダル自動展開 → のし・同梱物すべて☑ → 「📷 納品書スキャン（梱包完了）」ボタン押下（または同名 onclick） → ✓ 梱包完了トースト → 印刷／完了画面 → 待機画面に戻り右上「QRラベル再発行」→「（デモ）スキャン」で伝票確定 → 発行。
